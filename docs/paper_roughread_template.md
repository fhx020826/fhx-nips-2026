# continuous VLN 论文粗读模板与批量调研协议

## 1. 文档目的

本文档用于统一后续 `continuous VLN / VLN-CE` 论文粗读输出格式，服务于以下目标：

- 让粗读结果可直接横向比较，而不是每篇都按不同口径写
- 让后续 `baseline / codebase reconnaissance` 可以直接复用前面的粗读信息
- 让后续精读、复现优先级排序、方法设计时能快速回看关键信息
- 让后续批量化、半自动化调研有稳定字段可抽取，而不是完全依赖人工自由发挥

本文档是“粗读模板 + 填写说明 + 批处理协议”的统一版本，当前阶段先作为标准模板使用，不在这里填写具体论文内容。

---

## 2. 适用范围

优先适用于以下论文：

- `continuous VLN / VLN-CE` 主任务论文
- `R2R-CE / RxR-CE` 主 benchmark 论文
- 与 continuous VLN 强相关的 challenge / benchmark / dataset 扩展论文
- 对后续 `baseline / codebase reconnaissance` 有直接价值的代表性系统论文

默认不用于：

- 纯离散 VLN 上游方法
- 泛 embodied navigation 但不以 continuous VLN 为主任务的论文
- survey / position paper

---

## 3. 粗读输出原则

1. 只写已核实的信息；如果是推断，必须明确写成“当前判断”。
2. 方法描述优先回答“它解决了什么接口问题”，而不只是复述模块名。
3. 实验描述优先回答“能不能和我们的目标 benchmark 直接比较”。
4. 对每篇论文都要给出“是否值得继续投入”的判断，避免只做摘要不做决策。
5. 每篇论文都尽量保留可追溯证据入口，便于后续复查。

## 5. 每个部分应该写什么

### 5.1 基本信息

这一部分不是为了“资料卡完整”，而是为了后续少回头查。

应重点回答：

- 这篇论文是不是 strict 主线论文
- 它什么时候公开，时间线上处于什么位置
- 它到底有没有代码、项目页、模型、数据入口
- 它的录用和发布信息是否已经核实

相比只写标题、作者、日期，这里新增了：

- `任务身份`
- `项目 / 代码 / 模型 / 数据`
- `证据来源`

这三项对后续批量侦察最重要。

### 5.2 这篇论文要解决什么问题

这一部分不等于摘要翻译。

应重点写：

- 作者认为 prior work 的根本问题是什么
- 这篇论文试图修补的是哪个“结构性缺口”
- 这个缺口和 continuous VLN 的关系是什么

建议尽量用“问题 - 缺口 - 重要性”的逻辑来写。

### 5.3 一句话概括方法

这一部分的作用是快速回忆。

要求：

- 一句话能说清“谁负责什么”
- 避免只写模型名堆叠
- 尽量把高层目标和低层执行接口点说出来

### 5.4 核心方法

这一部分是粗读模板里最核心的主体。

建议始终围绕以下几个问题展开：

- 整个系统的输入输出是什么
- 高层和低层是怎么分工的
- 历史、记忆、进度、subgoal、控制等信息是怎么传递的
- 它相比 prior work 真正不同在哪

新增 `与 prior work 的本质区别`，是为了防止写成“模块流水账”。

### 5.5 实验做了什么，结果如何

这里不追求把所有表都复述一遍，而是优先回答：

- 它测了哪些 benchmark
- 设置能不能和我们的目标 benchmark 直接比较
- 主表里真正值得记住的结果是什么
- 消融到底支持了什么方法学判断

这里建议优先写：

- 主 benchmark
- 主 split
- 主指标
- 最相关 baseline 的对比

### 5.6 benchmark 可比性与复现生态

这是你原模板里最值得补强的一块。

因为后续你不只是要“读懂论文”，还要做：

- baseline 选型
- codebase reconnaissance
- 方法公平比较

所以这里新增了两个明确模块：

- `可比性检查`
- `复现生态`

应重点检查：

- 是否用了额外数据
- 是否用了额外监督
- 是否用了额外传感器
- 是否含 ensemble / test-time recipe
- 是否真的公开代码与 checkpoint
- 环境是否老旧到难以复现

### 5.7 亮点

亮点不等于“作者 claimed contributions”。

建议写：

- 这篇论文最值得记住的结构判断
- 最值得后续借鉴的接口设计
- 最能影响你课题方向的地方

通常写 3 条足够。

### 5.8 局限与风险

这一部分的目的不是吹毛求疵，而是避免后续误用。

建议区分：

- 论文已显式暴露的限制
- 你基于方法和实验能看出的潜在问题

尤其要关注：

- benchmark 是否狭窄
- 代价是否很高
- 可扩展性是否弱
- 真实部署或闭环稳定性是否不足

### 5.9 对当前课题的启发

这是你原模板里已经很强的一部分，建议保留并进一步结构化。

新增 `它对应我们的哪个核心问题`，目的是把每篇论文都映射到当前已经明确的研究主轴：

- history / memory
- progress
- hierarchical planning-control
- subgoal / latent bridge
- obstacle avoidance
- deadlock recovery
- closed-loop stability

这样后续回看时，不会只记得“这篇不错”，而是知道“它具体对哪条主线有用”。

### 5.10 是否值得继续投入

这是面向决策的新模块。

粗读如果没有这一步，很容易变成“写了很多，但不知道下一步该干嘛”。

建议每篇论文都给出：

- 是否值得精读
- 是否值得优先复现 / 侦察代码
- 后续具体动作建议

### 5.11 一句话评价

这一部分的作用是帮助后续快速检索记忆。

要求：

- 尽量像“论文定位标签”
- 不是感想，而是结论

## 6. 你的原模板 vs 优化模板

你的原模板已经非常强，尤其强在：

- 问题意识很明确
- 方法拆解足够细
- 实验与启发写得很到位
- 已经不只是摘要，而是在做研究判断

本次优化不是推翻，而是往“可比较、可决策、可批量化”方向补强。

### 6.1 保留不动的核心骨架

保留了你原本最重要的 7 大核心段落：

- 基本信息
- 论文要解决什么问题
- 核心方法
- 实验结论
- 亮点
- 局限
- 对你课题的启发

### 6.2 主要增强点

新增的关键模块有：

1. `任务身份`
   - 解决“这篇是否属于 strict 主线”的快速判断问题

2. `证据来源`
   - 解决后续复查困难和信息漂移问题

3. `benchmark 可比性检查`
   - 解决不同论文结果不能直接横比的问题

4. `复现生态`
   - 解决“论文很好，但代码难跑”的现实落地问题

5. `与 prior work 的本质区别`
   - 解决粗读容易写成模块堆砌的问题

6. `是否值得继续投入`
   - 解决粗读后没有行动结论的问题

7. `核心问题映射`
   - 解决后续方法设计时无法快速回收已有阅读资产的问题

### 6.3 为什么这个版本更适合批量调研

因为这个版本把自由叙述和结构化字段结合起来了：

- 上层仍然保留研究者视角的自然语言判断
- 下层加入了稳定可抽取的结构字段

这样后续无论是：

- 人工批量写 5 篇一组
- 半自动抓 paper/project/code/venue 信息
- 后续整理成表格或数据库

都更容易衔接。

其中 `interface_tags` 建议从以下集合里选：

- `subgoal-bridge`
- `obstacle-avoidance`
- `deadlock-recovery`
- `closed-loop-stability`
- `zero-shot-transfer`
- `real-world-deployment`

## 8. 推荐的 5 篇一批处理流程

后续建议始终按 5 篇一组推进。

### Step 1：先统一拉取事实层信息

先补齐每篇的：

- arXiv
- venue / OpenReview / publisher
- project
- code
- model
- data

这一轮只做事实核查，不急着写长文。

### Step 2：再统一做任务身份与可比性判断

这一轮只回答：

- 是否 strict 主线
- 是否直接可比 `R2R-CE / RxR-CE`
- 是否用了额外数据 / 额外监督 / 额外传感器

### Step 3：再写方法与实验主体

这一轮才开始写：

- 问题定义
- 方法骨架
- 关键实验
- 消融与分析

### Step 4：最后写研究判断

最后补：

- 亮点
- 局限
- 对当前课题的启发
- 是否值得继续投入

这样做的好处是：

- 不容易因为前面事实层没查清楚而反复返工
- 很适合批量并行推进
- 更适合脚本辅助

## 9. 后续落地建议

后续真正开始粗读时，建议采用两层产物：

1. `论文卡片文档`
   - 按 5 篇一批输出完整粗读

2. `结构化汇总表`
   - 只记录可比性、复现生态、优先级等短字段

前者负责研究理解，后者负责决策与检索。

---

## 10. 推荐使用方式

建议后续每次开始新一批粗读时，都默认遵循以下优先级：

1. 先从 `docs/paper_quality_shortlist.md` 中选 5 篇
2. 按本文模板逐篇写卡片
3. 每篇必须给出“是否值得继续投入”
4. 每批结束后，再决定：
   - 继续下一批粗读
   - 转入 baseline / codebase reconnaissance
   - 或挑最关键 1–2 篇进入精读

这个模板的目标不是把每篇都写成长篇综述，而是尽快形成：

- 可比较
- 可复查
- 可决策
- 可自动化扩展

的论文粗读资产。
