# Coding 规范

## 目标

本项目所有新增或修改代码都必须遵守统一规范，保证：

- 可读
- 可复现
- 易调试
- 易清理
- 中文注释清晰

## 通用原则

1. 先定义问题，再写代码。
2. 先最小可验证，再逐步扩展。
3. 优先复用明确可靠的模块，不做无根据的大而全封装。
4. 单个文件职责要明确，避免混杂脚本、实验逻辑、数据处理和评测逻辑。
5. 所有实验相关代码都必须能说明输入、输出、依赖和运行命令。

## 目录约定

- `src/`: 核心源码
- `scripts/`: 可直接运行的脚本
- `configs/`: 配置文件
- `docs/`: 计划、调研、说明文档
- `data/`: 本地数据，不上传远程
- `artifacts/`: 中间结果、日志、可清理产物，不上传远程
- `tmp/`: 临时文件

## Python 规范

1. 默认 Python 3.10。
2. 优先写小函数，少写超长脚本。
3. 函数名、变量名语义明确，避免 `tmp`, `foo`, `x1` 这类命名。
4. 能加类型标注就加类型标注。
5. 严禁静默吞掉异常；出错时要给出可定位的信息。
6. 配置与硬编码分离；路径、阈值、超参数尽量放入配置文件。

## 注释规范

1. 注释统一用中文。
2. 只解释“为什么这样做”或“不直观的实现细节”。
3. 不写废话式注释，例如“给变量赋值”“循环开始”。

## 脚本规范

每个可执行脚本都应尽量包含：

- 功能说明
- 必要参数
- 输入输出说明
- 示例命令

## 实验规范

1. 每次实验必须记录：
   - 目的
   - 代码版本
   - 配置
   - 关键结果
   - 是否保留
2. 明确失败且无参考价值的中间结果，可以在告知用户后清理。
3. 大日志、数据集、模型权重、缓存默认不上传远程仓库。

## Git 规范

1. 重要代码和文档更新必须提交。
2. 提交信息简洁明确，推荐：
   - `docs: ...`
   - `feat: ...`
   - `fix: ...`
   - `refactor: ...`
   - `research: ...`
3. 未验证通过的改动不要宣称完成。

## 代理与环境规范

1. 任何访问外网的命令都优先通过代理执行。
2. 默认使用项目专属 conda 环境：`fhx-nips-2026`。
3. 运行外网相关命令时，优先使用：

```bash
bash -ic 'proxy >/dev/null; <你的命令>'
```

## 清理原则

1. 每轮对话结束时检查是否有可清理文件。
2. 不直接删除可能还有价值的结果，先告知用户。
3. 清理目标优先级：
   - 已确认失败且无参考价值的输出
   - 重复日志
   - 过时临时文件
   - 多余草稿文档
