# Coding 规范

## 目标

本项目所有新增或修改代码都必须遵守统一规范，保证：
- 可读
- 可复现
- 易调试
- 易清理
- 中文注释清晰
- 目录长期保持干净整洁

## 工作区边界

1. 所有项目相关操作只在 `/home/hxfeng/fhx-nips-2026` 内进行。
2. 不把实验代码、草稿脚本、临时下载结果散落到仓库外。
3. 任何新的脚本、配置或文档都要放到明确目录中，避免顶层持续膨胀。

## 通用原则

1. 先定义问题，再写代码。
2. 先最小可验证，再逐步扩展。
3. 优先复用明确可靠的模块，不做无根据的大而全封装。
4. 单个文件职责要明确，避免混杂脚本、实验逻辑、数据处理和评测逻辑。
5. 所有实验相关代码都必须能说明输入、输出、依赖和运行命令。
6. 遇到报错必须定位并解决，不能把未解决错误当作“已完成”。

## 目录约定

- `src/`: 核心源码
- `scripts/`: 可直接运行的脚本
- `configs/`: 配置文件
- `docs/`: 计划、调研、说明文档
- `data/`: 本地数据，不上传远程
- `artifacts/`: 中间结果、日志、可清理产物，不上传远程
- `tmp/`: 临时文件
- `tmp_papers/`: 论文下载与文本抽取缓存，不上传远程
- `tmp_repos/`: 临时克隆仓库，不上传远程

## Python 与环境规范

1. 默认 Python 3.10。
2. 所有 Python 代码运行都应使用项目专属 conda 环境：`fhx-nips-2026`。
3. 若当前 shell 中 `conda` 不可用，先解决环境初始化问题，再继续执行 Python 任务。
4. 优先写小函数，少写超长脚本。
5. 函数名、变量名语义明确，避免 `tmp`, `foo`, `x1` 这类命名。
6. 能加类型标注就加类型标注。
7. 严禁静默吞掉异常；出错时要给出可定位的信息。
8. 配置与硬编码分离；路径、阈值、超参数尽量放入配置文件。

## 注释规范

1. 注释统一用中文。
2. 只解释“为什么这样做”或“不直观的实现细节”。
3. 不写废话式注释，例如“给变量赋值”“循环开始”。

## 脚本规范

每个可执行脚本都应尽量包含：
- 功能说明
- 必要参数
- 输入输出说明
- 示例命令

## 文档同步规范

1. 代码结构或项目状态发生变化时，要同步检查：
   - `project.md`
   - `work.md`
   - `user.md`
2. 论文调研流程发生变化时，要同步检查相关 `docs/` 文档是否过时。
3. 不只追加新内容，也要修正旧内容中已经过时或错误的部分。

## 实验规范

1. 每次实验必须记录：
   - 目的
   - 代码版本
   - 配置
   - 关键结果
   - 是否保留
2. 明确失败且无参考价值的中间结果，可以在告知用户后清理。
3. 大日志、数据集、模型权重、缓存默认不上传远程仓库。

## Git 规范

1. 重要代码和文档更新必须提交并推送到远程仓库。
2. 提交信息简洁明确，推荐：
   - `docs: ...`
   - `feat: ...`
   - `fix: ...`
   - `refactor: ...`
   - `research: ...`
3. 未验证通过的改动不要宣称完成。
4. 推送前确认不把大文件、日志、数据集、权重和临时缓存带入远程。

## 代理规范

1. 任何访问外网的命令都优先通过代理执行。
2. 读取 GitHub 仓库、README、release、issues 也视作外网访问。
3. 运行外网相关命令时，优先使用：

```bash
bash -ic 'proxy >/dev/null; <你的命令>'
```

4. 如果代理异常，先停止外网检索并及时通知用户修复。

## 清理原则

1. 每轮对话结束时检查是否有可清理文件。
2. 不直接删除可能还有价值的结果，先告知用户。
3. 清理目标优先级：
   - 已确认失败且无参考价值的输出
   - 重复日志
   - 过时临时文件
   - 多余草稿文档
4. `tmp/`、`tmp_papers/`、`tmp_repos/` 默认视为可清理候选区域，但删除前仍需先告知用户。
